How we estimate the number of Mutual Economic State Transitions generated by each transaction category.

1. What We Are Estimating

For each of the 29 transaction categories in the Universe of Finance, we estimate a MEST multiplier: the average number of bilateral state changes generated per transaction.

    MEST/s  =  Category TPS  x  MEST Multiplier

    Total MEST/s  =  SUM across all 29 categories

Unlike the Big Number (which de-duplicates overlapping TPS), the MEST Number does not de-duplicate. Each MEST is a unique bilateral state change between a unique pair of parties. Two MESTs from the same transaction are not duplicates — they are distinct accounting events at different points in the cascade.

Stage-by-Stage MEST Identification

For each stage, we ask:

  1. Who are the two parties? Both must have a material interest in the accuracy of the record.
  2. What value changed state? A balance, ownership, obligation, entitlement, or encumbrance must have changed.
  3. Is it material? Internal bookkeeping within one entity (e.g., a bank moving funds between its own sub-ledgers) is excluded.

3. Estimation Process

Step 1: Map the Transaction Lifecycle

For each category, we map the specific lifecycle stages. Example for Consumer Cards:

    Consumer Card Purchase — Lifecycle Map
    ═══════════════════════════════════════

    [Cardholder taps card]
         │
         ▼
    ┌─ INITIATION ──────────────────────────────────────┐
    │  Merchant terminal sends auth request to acquirer  │
    │  No MEST here (single-party action)                │
    └─────────────────────┬──────────────────────────────┘
                          ▼
    ┌─ EXECUTION ───────────────────────────────────────┐
    │  Issuer evaluates and places hold on account       │
    │  MEST #1: Cardholder + Issuer (balance reserved)   │
    └─────────────────────┬──────────────────────────────┘
                          ▼
    ┌─ CLEARING ────────────────────────────────────────┐
    │  Network matches presentment to authorization      │
    │  MEST #2: Acquirer + Network (obligation created)  │
    │  MEST #3: Issuer + Acquirer (interchange calc)     │
    └─────────────────────┬──────────────────────────────┘
                          ▼
    ┌─ SETTLEMENT ──────────────────────────────────────┐
    │  Issuer pays network (net of interchange)          │
    │  MEST #4: Issuer + Network (funds transferred)     │
    │  Network pays acquirer                             │
    │  MEST #5: Network + Acquirer (funds transferred)   │
    └─────────────────────┬──────────────────────────────┘
                          ▼
    ┌─ POST-SETTLEMENT ─────────────────────────────────┐
    │  Acquirer credits merchant bank account             │
    │  MEST #6: Merchant + Acquirer (funds received)     │
    │  Issuer posts to cardholder statement + rewards     │
    │  MEST #7: Cardholder + Issuer (balance updated)    │
    └───────────────────────────────────────────────────┘

    Total: 7 MESTs per card purchase

Step 2: Count Bilateral State Changes

For each lifecycle stage, count the number of distinct bilateral state changes. The rules:

  • Each unique pair of parties counts once per state change type. If the issuer and acquirer exchange both securities information and cash, that is 2 MESTs.
  • Netting reduces MEST count. If 1000 transactions between the same two parties are netted to 1 settlement, the settlement is 1 MEST (but the 1000 individual clearing entries are still 1000 MESTs each).
  • Internal operations are excluded. A bank's own risk system updating an internal score is not a MEST. A bank posting margin to a CCP is a MEST (two parties).

Step 3: Estimate the Average

Most categories have a range of transaction types with different cascade depths. We estimate a weighted average:

    Category Average MEST = SUM(subcategory_share x subcategory_MEST_multiplier)

    Example — Bank Transfers:
    ├─ ACH/BACS batch (60% of volume):     4 MESTs
    ├─ Real-time payments (30%):            5 MESTs
    └─ Wire transfers (10%):               7 MESTs

    Weighted average: 0.6(4) + 0.3(5) + 0.1(7) = 4.6 ≈ 5 MESTs

Step 4: Cross-Check With Industry Knowledge

We validate multiplier estimates against: - Known processing step counts from payment scheme documentation - Settlement flow diagrams from clearing houses (CLS, DTCC, LCH) - Card network four-party model documentation - BIS CPMI reports on payment system architectures

4. MEST Counting Rules

What Counts as a MEST

Criterion Counts Does Not Count
Balance change Bank credits merchant account Bank updates internal P&L sub-ledger
Ownership transfer CSD transfers securities Nominee re-registration (same beneficial owner)
Obligation created CCP novates trade Trader enters order (no counterparty yet)
Regulatory record Trade report to FINRA Internal compliance flag
Fee/commission Interchange allocated between issuer + acquirer Internal cost allocation
Collateral movement Margin posted to CCP CCP recalculates own risk metrics

Edge Cases

  1. Netting: If N transactions are netted to 1 settlement, we count N clearing MESTs + 1 settlement MEST (not N settlement MESTs).

  2. Real-time systems: When clearing and settlement are atomic (e.g., UPI), we still count the clearing MEST and settlement MEST separately because both banks independently update their ledgers.

  3. On-chain transactions: A blockchain transfer is 2 MESTs (sender balance change + receiver balance change). Validator consensus is internal to the network, not a MEST. Gas/fee payment to validators is a separate MEST.

  4. DeFi composability: A flash loan that routes through 3 pools is 3 on-chain transactions (each already counted in L1/L2 TPS). We do not multiply L1/L2 MESTs by DeFi composability — the TPS already reflects the on-chain volume.

  5. Recurring/scheduled transactions: Each execution is a separate MEST cascade. Setting up the schedule is not a MEST (no value moves).

Aggregate Confidence

The MEST Number (~545,000 MEST/s) has Medium confidence overall:

  • The top 5 categories (88% of MESTs) are Medium-to-High confidence
  • The weighted-average multiplier (7.4x) is plausible against industry benchmarks (Visa's four-party model alone implies 6-7 processing steps per card transaction)
  • The range (410K-720K MEST/s) captures estimation uncertainty

6. Limitations

What We Cannot Observe

  1. Internal bookkeeping MESTs: When a bank's treasury desk moves funds between internal accounts, that may or may not be a MEST (depends on whether an external party cares). We conservatively exclude all internal movements.

  2. Shadow cascades: Some transactions trigger downstream events we cannot map — e.g., a merchant receiving a card payment may trigger an automated inventory reorder, which triggers a supplier payment, which triggers a whole new MEST cascade. We count only the direct cascade of the original transaction.

  3. Informal economy: Cash transactions, hawala, barter, and unrecorded economic activity generate MESTs (buyer + seller both know what happened) but are unobservable. The MEST Number covers only the formal, recorded economy.

  4. Micro-MESTs in reporting: Every transaction eventually appears on a bank statement, a tax return, and potentially multiple regulatory reports. We count the primary reporting MESTs (statement line, regulatory trade report) but not secondary cascades (auditor review, tax filing, investor reporting).

What This Means

The MEST Number (~545,000/s) is likely a lower bound on the true bilateral state-change volume of the global economy. Including shadow cascades, secondary reporting, and the informal economy could push the real number above 1 million MEST/s. We prefer a conservative, defensible estimate.

Relationship to the Big Number

Metric Value What It Measures
The Big Number ~73,750 TPS User-visible economic events per second
The MEST Number ~545,000 MEST/s Bilateral state changes per second
Ratio ~7.4x Average cascade depth

The two numbers are complementary: - The Big Number is the right metric for economic activity volume - The MEST Number is the right metric for infrastructure burden - The ratio is useful for comparing institutional complexity across transaction types

7. Methodology Validation

Cross-Checks Applied

  1. Card network processing steps: Visa's documentation describes 6-8 processing steps per transaction. Our 7x estimate falls in this range.

  2. DTCC settlement statistics: DTCC processes ~$2.4 quadrillion/year in securities. With ~44 TPS in securities settlement and a 4x MEST multiplier, that implies ~176 settlement MESTs/s across the CSD layer. This is consistent with DTCC's reported daily instruction volumes.

  3. CLS settlement volumes: CLS settles ~$6.6 trillion/day in FX. With ~127 TPS in forex and 8x MEST multiplier, that implies ~1,016 forex MESTs/s. CLS processes ~1 million payment instructions/day, which aligns (~12 MEST/s just for CLS netting outputs, plus bilateral settlement legs).

  4. Blockchain transparency: L1/L2 at 2x is directly verifiable — every on-chain transfer has exactly one sender and one receiver state change. This is the one category where the MEST multiplier is known with certainty.

8. Future Improvements

  1. Primary source validation: Survey clearing house and CSD documentation to validate or refine multipliers for the top 10 categories.

  2. Netting ratio refinement: The current model does not explicitly model netting ratios (e.g., CCP multilateral netting typically reduces settlement MESTs by 90-95%). More granular netting data would sharpen the estimate.

  3. Time-series: As the MEST framework matures, build a 2015-2025 MEST time-series alongside the TPS time-series. Hypothesis: MEST growth is slower than TPS growth because real-time settlement compresses cascade depth.

  4. Sector-specific deep dives: Produce detailed MEST lifecycle maps for the top 5 categories (cards, wallets, bank transfers, ETD, equities) with primary source validation.
